Question 1: The 'burbs

This question is worth 50% of the assignment mark.

Write a P3D/OpenGL program that draws a neighbourhood of similar-but-different looking houses by creating one parameterized house and then redrawing it multiple times with different parameter values. The entire house is drawn in one coordinate system and transformed for each copy. The components of a house are each drawn in their own coordinate systems and arranged within the house.

Your house must be made up of the following components, which in turn are made up of the other components shown underneath:

  • roof
    • rooftop
    • chimney
  • front wall
    • bricks/logs/pillars/etc (in rows and/or columns)
    • door
    • window
  • tree
    • leaves
    • trunk

The bottom-level components must each be drawn in their own coordinate system, and must be made up of at least three different primitives with different and clearly-visible colours. The higher-level components must in turn be drawn in their own coordinate system, with the bottom-level elements arranged within.

Each of the three higher-level components must be separately and non-uniformly scaleable; e.g. wide front, tall roof, tiny tree. When the front wall is scaled, the repeated bottom-level component (bricks/logs/pillars/etc) must keep their original proportions; e.g. a tall wall will have more rows of bricks, not tall bricks. The other bottom-level components can either be scaled relatively or separately.

In addition to the higher-level scaling, you must implement each of the following parameterized transformations for at least one component of the bottom-level geometry, and at least one component of the higher-level geometry:

  • translation
  • rotation
  • scaling (already done for the higher-level)
  • reflection
  • shearing

Changing these parameters within a restricted range should maintain the "structural integrity" of the components; e.g. a small rotation shouldn't cause the roof to come off of the house.

For your final output, draw the house nine times. The version in the centre of the screen should be "normal": uniform scale, no rotation, etc. The other eight copies should each have different values for the parameters, as well as a variety of uniform sizes and rotations for the entire drawing. Try to keep them from overlapping.

In your readme file, for each parameterized transformation, list the names of the components that you applied it to, and for each copy of the house drawn, list the different parameters and transformations you used, in words. For example, your readme file should contain something formatted like this:

Question 1 transformations:

translation: tree, window
rotation: tree, chimney
scaling: ...
...

top-left copy: rotated 8 degrees, scaled 75%
 - tree translated left, window translated up
 - tree rotated -5 degrees, chimney rotated 5 degrees
 - wall has 4 rows of 5 bricks
 - ...
...

Question 2: Triangle composer

This question is worth 50% of the assignment mark.

Write an interactive OpenGL program that will allow you to create and edit 2D triangles, which can be connected by shared vertices.

When you first start your program, it should display a colour selection panel and an empty drawing area. One colour will be selected by default. Clicking in the colour selection panel will select a different colour. Clicking in the drawing area will specify the first vertex for a triangle. The next two clicks will specify the remaining two vertices and close the triangle, drawing it as a filled shape using the currently selected colour. This process can be repeated to draw additional triangles. If you click on or close enough to an existing vertex (gravity), the triangle can share one or more vertices with existing triangles.

You can also click in an existing triangle to select it. The selected triangle will be drawn distinctively (e.g. thicker outline, vertices drawn as circles). Clicking and dragging the triangle will move it. Clicking and dragging a vertex of the selected triangle will move it. The vertex will be moved for all triangles that share the vertex. If a triangle is selected, clicking in an empty space in the drawing area will deselect it.

It is also possible to transform the view. You can use the keyboard to translate and scale the view (not the vertices!). If a triangle is selected, the same keyboard keys will translate and scale the selected triangle around its centre. You can also use the keyboard to rotate the selected triangle around its centre. The rotation transformation does not apply to the view.

  • Use ten different colours in the colour panel at the bottom of the window. They should be placed in equally-sized rectangles.
  • The colour panel should scale to fill the width of the window.
  • Indicate the currently-selected colour somehow, such as with an outline around it.
  • Draw each triangle as a filled shape with a white outline.
  • Use some visually-distinct colour for the selected triangle.
  • Use 7 pixels as a "gravity" distance (radius) for connecting vertices.
  • Use w a s d keys for translating the view or the selected triangle.
  • Use c v keys for scaling the view or the selected triangle. One will increase the scale by 0.1, the other will decrease it by 0.1.
  • Use the z key to restore the view to its original state (no translation or scaling). It has no effect on triangles.
  • Use e r keys for rotating the selected triangle. One will rotate clockwise, the other counter-clockwise. Rotate by 3.75 degrees (PI/48 radians) with each press. It has no effect on the view.
  • Calculate the triangle centre as the average of its x and y coordinates.
  • Triangle creation, selection, and triangle/vertex dragging should still work in a transformed view.
  • Keyboard commands do not need to work while you are creating or dragging triangles or vertices.
  • Optionally: When you are transforming a triangle, all triangles that share vertices with it are "connected", and all triangles that share vertices with other connected triangles are also connected. Transform all connected triangles together. To determine connected triangles, you can search repeatedly for each shared index in other triangles.
  • Transforming triangles and vertices can change their geometry.
  • Transforming the view should not change the vertex data; vertices should be transformed before drawing.
  • Use floating-point coordinates for triangle vertices, regardless of your chosen coordinate system.
  • Don't put any special effort into managing "in front of" and "behind". This may cause strange artefacts, such as (parts of) a selected triangle appearing behind others.
  • Behave as well as you can. There are a lot of possible states during input, and managing them is difficult. A "bit" of glitchiness is acceptable (but not too much).
  • Crash under no circumstances.

To enable window resizing, add a call surface.setResizable(true); to your setup() method.
